# Prompt per Integrazione SonarQube - Spring Boot Application

## 1. OBIETTIVO GENERALE

Integrare SonarQube per l'analisi statica del codice nell'applicazione Spring Boot esistente, configurando:
- Plugin Maven per SonarQube
- Script PowerShell automatizzato per eseguire l'analisi
- Proprietà di configurazione per connessione al server SonarQube
- Best practices per qualità del codice e code coverage

---

## 2. CONFIGURAZIONE MAVEN (pom.xml)

### 2.1 Plugin SonarQube Maven

Aggiungere nella sezione `<build><plugins>` del file `pom.xml`:

```xml
<plugin>
    <groupId>org.sonarsource.scanner.maven</groupId>
    <artifactId>sonar-maven-plugin</artifactId>
    <version>4.0.0.4121</version>
</plugin>
```

### 2.2 Plugin JaCoCo per Code Coverage

Aggiungere il plugin JaCoCo per generare report di copertura del codice:

```xml
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <version>0.8.11</version>
    <executions>
        <execution>
            <id>prepare-agent</id>
            <goals>
                <goal>prepare-agent</goal>
            </goals>
        </execution>
        <execution>
            <id>report</id>
            <phase>test</phase>
            <goals>
                <goal>report</goal>
            </goals>
        </execution>
        <execution>
            <id>jacoco-check</id>
            <goals>
                <goal>check</goal>
            </goals>
            <configuration>
                <rules>
                    <rule>
                        <element>PACKAGE</element>
                        <limits>
                            <limit>
                                <counter>LINE</counter>
                                <value>COVEREDRATIO</value>
                                <minimum>0.50</minimum>
                            </limit>
                        </limits>
                    </rule>
                </rules>
            </configuration>
        </execution>
    </executions>
</plugin>
```

### 2.3 Proprietà SonarQube

Aggiungere nella sezione `<properties>` del pom.xml:

```xml
<!-- SonarQube Configuration -->
<sonar.host.url>http://localhost:9000</sonar.host.url>
<sonar.projectKey>corso-ntt-38</sonar.projectKey>
<sonar.projectName>Corso NTT 38 - Spring Boot Application</sonar.projectName>
<sonar.projectVersion>${project.version}</sonar.projectVersion>
<sonar.java.source>21</sonar.java.source>
<sonar.language>java</sonar.language>
<sonar.sourceEncoding>UTF-8</sonar.sourceEncoding>

<!-- Percorsi sorgenti e test -->
<sonar.sources>src/main/java,src/main/resources</sonar.sources>
<sonar.tests>src/test/java</sonar.tests>

<!-- JaCoCo Code Coverage -->
<sonar.java.coveragePlugin>jacoco</sonar.java.coveragePlugin>
<sonar.coverage.jacoco.xmlReportPaths>target/site/jacoco/jacoco.xml</sonar.coverage.jacoco.xmlReportPaths>

<!-- Esclusioni analisi -->
<sonar.exclusions>
    **/target/**,
    **/*.js,
    **/*.css,
    **/static/**,
    **/templates/**
</sonar.exclusions>
<sonar.test.exclusions>
    **/test/**,
    **/*Test.java,
    **/*Tests.java
</sonar.test.exclusions>

<!-- Quality Gate -->
<sonar.qualitygate.wait>false</sonar.qualitygate.wait>
```

### 2.4 Proprietà Opzionali per Autenticazione

Se il server SonarQube richiede autenticazione:

```xml
<!-- Autenticazione (opzionale) -->
<sonar.login>${env.SONAR_TOKEN}</sonar.login>
<!-- oppure -->
<sonar.token>${env.SONAR_TOKEN}</sonar.token>
```

**Nota**: Il token deve essere generato dall'interfaccia SonarQube (User > My Account > Security > Generate Tokens)

## 3. SCRIPT POWERSHELL (scripts/run-sonar-analysis.ps1)

### 3.1 Requisiti Script

Creare uno script PowerShell nella cartella `scripts/` con le seguenti caratteristiche:

1. **Nome file**: `run-sonar-analysis.ps1`
2. **Percorso**: `scripts/run-sonar-analysis.ps1`
3. **Funzionalità**:
   - Eseguire il comando Maven per l'analisi SonarQube
   - Gestire errori e fornire feedback chiaro
   - Supportare parametri opzionali (token, host URL, project key)
   - Verificare prerequisiti (Maven, Java, server SonarQube disponibile)
   - Generare log di esecuzione
   - Fornire riepilogo finale con link al dashboard SonarQube

### 3.2 Struttura Script Richiesta

```powershell
<#
.SYNOPSIS
    Esegue l'analisi del codice con SonarQube tramite Maven

.DESCRIPTION
    Script per automatizzare l'esecuzione dell'analisi statica del codice
    utilizzando SonarQube. Esegue test, genera report di coverage con JaCoCo
    e invia i risultati al server SonarQube.

.PARAMETER SonarHostUrl
    URL del server SonarQube (default: http://localhost:9000)

.PARAMETER SonarToken
    Token di autenticazione SonarQube (opzionale)

.PARAMETER ProjectKey
    Chiave del progetto SonarQube (default: corso-ntt-38)

.PARAMETER SkipTests
    Se presente, salta l'esecuzione dei test (non consigliato)

.PARAMETER Clean
    Se presente, esegue mvn clean prima dell'analisi

.EXAMPLE
    .\run-sonar-analysis.ps1
    
.EXAMPLE
    .\run-sonar-analysis.ps1 -SonarToken "sqp_123456789abcdef"
    
.EXAMPLE
    .\run-sonar-analysis.ps1 -SonarHostUrl "http://sonar.company.com:9000" -Clean

.NOTES
    Author: Corso NTT 38
    Requires: Maven, Java 21, SonarQube Server
#>
```

### 3.3 Funzionalità Richieste

1. **Validazione prerequisiti**:
   - Verificare che Maven sia installato (`mvn --version`)
   - Verificare che Java 21 sia disponibile
   - Testare connettività al server SonarQube (HTTP GET su `/api/system/status`)

2. **Parametri supportati**:
   - `-SonarHostUrl` (URL server, default: http://localhost:9000)
   - `-SonarToken` (token autenticazione, opzionale)
   - `-ProjectKey` (chiave progetto, default: corso-ntt-38)
   - `-SkipTests` (switch per saltare test)
   - `-Clean` (switch per eseguire clean)
   - `-Verbose` (output dettagliato)

3. **Gestione errori**:
   - Try-catch per ogni operazione critica
   - Exit code significativi (0 = successo, 1 = errore prerequisiti, 2 = errore build, 3 = errore analisi)
   - Messaggi colorati (verde = successo, rosso = errore, giallo = warning)

4. **Output informativo**:
   - Banner iniziale con informazioni script
   - Progress indicators durante esecuzione
   - Riepilogo finale con metriche chiave
   - Link diretto al dashboard SonarQube del progetto

5. **Sequenza comandi Maven**:
```powershell
# Pulizia (opzionale)
mvn clean

# Compilazione e test con coverage
mvn verify

# Analisi SonarQube
mvn sonar:sonar `
  -Dsonar.host.url=$SonarHostUrl `
  -Dsonar.projectKey=$ProjectKey `
  [-Dsonar.token=$SonarToken]
```

### 3.4 Logging

Lo script deve creare un file di log nella cartella `logs/`:
- Nome: `sonar-analysis-YYYYMMDD-HHmmss.log`
- Contenuto: timestamp, comandi eseguiti, output Maven, errori
- Rotazione: mantenere ultimi 10 log

## 6. BEST PRACTICES

### 6.1 Quality Rules per Spring Boot

Configurare regole SonarQube specifiche per Spring Boot:
- **Security**: SQL Injection, XSS, CSRF protection
- **Spring**: `@Autowired` su costruttori, gestione `@Transactional`
- **REST**: DTO validation, HTTP status codes appropriati
- **Exception Handling**: `@ControllerAdvice` per gestione centralizzata errori

### 6.2 Integrazione CI/CD

Pipeline tipica:
```yaml
# Esempio GitHub Actions / Azure DevOps
1. Checkout codice
2. Setup Java 21
3. Maven: mvn clean verify
4. SonarQube: mvn sonar:sonar -Dsonar.token=$SONAR_TOKEN
5. Quality Gate check (opzionale)
```

### 6.3 Gestione Eccezioni

Escludere dall'analisi:
- File generati automaticamente
- Configurazioni statiche (HTML, CSS, JS)
- Test di integrazione legacy
- Codice di terze parti

```xml
<sonar.exclusions>
    **/target/**,
    **/generated/**,
    **/*.js,
    **/*.css,
    **/static/**,
    **/templates/**,
    **/test-data/**
</sonar.exclusions>
```

### 6.4 Code Coverage Target

Obiettivi realistici:
- **Controllers**: 80%+ coverage
- **Services**: 85%+ coverage
- **Repositories**: 70%+ coverage (JPA auto-generated)
- **Models/DTOs**: 60%+ coverage (getter/setter)
- **Configuration**: 50%+ coverage

### 6.5 Issue Severity Prioritization

1. **Blocker**: Fix immediato (Security Vulnerabilities critiche)
2. **Critical**: Fix entro 1 giorno (Bugs che causano crash)
3. **Major**: Fix entro 1 settimana (Code Smells che impattano maintainability)
4. **Minor**: Fix opportunistico (Miglioramenti minori)
5. **Info**: Revisione volontaria (Suggerimenti ottimizzazione)

## 7. TROUBLESHOOTING

### 7.1 Problemi Comuni

**Errore: "Sonar server 'http://localhost:9000' can not be reached"**
- Verificare che SonarQube sia avviato: `docker ps` o navigare a http://localhost:9000
- Controllare firewall/antivirus
- Provare con IP esplicito: `http://127.0.0.1:9000`

**Errore: "401 Unauthorized"**
- Generare token SonarQube valido
- Verificare token con: `curl -u YOUR_TOKEN: http://localhost:9000/api/authentication/validate`
- Usare `-Dsonar.token=` invece di `-Dsonar.login=` (versioni recenti)

**Code Coverage = 0%**
- Verificare che JaCoCo plugin sia configurato
- Eseguire `mvn test` prima di `mvn sonar:sonar`
- Usare `mvn clean verify sonar:sonar` (comando completo)
- Controllare percorso report: `target/site/jacoco/jacoco.xml`

**Errore: "Project key already exists"**
- Progetto già presente su SonarQube
- Opzioni: eliminare progetto esistente o usare projectKey diverso

### 7.2 Debug

Eseguire con output verbose:
```bash
mvn sonar:sonar -X -Dsonar.verbose=true
```

Verificare connessività:
```powershell
# Test HTTP
Invoke-RestMethod -Uri "http://localhost:9000/api/system/status"

# Output atteso:
# {"id": "...", "version": "...", "status": "UP"}
```
